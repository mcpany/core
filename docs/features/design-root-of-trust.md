# Design Doc: Root-of-Trust Attestation Engine
**Status:** Draft
**Created:** 2026-02-27

## 1. Context and Scope
The "Clinejection" and "OpenClaw UI" exploits demonstrate that the integrity of the tool supply chain is the weakest link in agentic systems. Simple SHA256 checks are insufficient when the registry itself or the local configuration can be compromised. MCP Any requires a "Root-of-Trust" (RoT) to ensure that only cryptographically verified tools are ever executed.

## 2. Goals & Non-Goals
* **Goals:**
    * Implement a multi-layered attestation system (Local Dev-Key, Corporate CA, or Hardware TPM).
    * Ensure that MCP server configurations cannot be modified without re-attestation.
    * Provide a "Secure Registry" where tools are indexed only after their provenance is verified.
* **Non-Goals:**
    * Encrypting tool outputs (this is a separate privacy concern).
    * Replacing existing MCP transport encryption (e.g., TLS).

## 3. Critical User Journey (CUJ)
* **User Persona:** Enterprise Security Architect.
* **Primary Goal:** Prevent "Shadow Tools" from being registered in the production MCP Any instance.
* **The Happy Path (Tasks):**
    1. The Architect configures MCP Any to require "Signed Attestations" for all new upstreams.
    2. A developer attempts to add a new local Python MCP server.
    3. MCP Any requests a signature of the server's manifest and executable.
    4. The developer provides a signature generated by the corporate build pipeline.
    5. MCP Any verifies the signature against the Root-of-Trust and allows the registration.
    6. If the local executable is later tampered with, MCP Any detects the signature mismatch and disables the tool.

## 4. Design & Architecture
* **System Flow:**
    - **Manifest Signing**: Tools include an `attestation.json` containing hashes of all files and a signature.
    - **Verification Loop**: At startup and periodic intervals, MCP Any re-verifies the hashes against the signature.
    - **Hardware Integration**: (Optional) Use local TPM (Trusted Platform Module) to store the Root-of-Trust keys.
* **APIs / Interfaces:**
    - `mcpserver.register(manifest, signature)`: Enhanced registration call.
    - `mcpany_status`: Includes an `attested: boolean` field for every tool.
* **Data Storage/State:** Signatures and public keys are stored in a protected SQLite database (`attestation.db`).

## 5. Alternatives Considered
* **Code Signing (OS Level)**: Relying on macOS/Windows code signing. *Rejected* because it doesn't cover script-based tools (Python/JS) effectively.
* **Centralized Store**: Requiring all tools to come from a single "App Store." *Rejected* as it breaks the decentralized nature of MCP.

## 6. Cross-Cutting Concerns
* **Security (Zero Trust):** If the Root-of-Trust key is compromised, the entire system is at risk. Support for hardware-backed keys is critical.
* **Observability:** Any attestation failure must trigger a high-priority alert in the Management UI and Audit Logs.
